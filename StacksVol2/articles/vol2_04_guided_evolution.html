The Guided Evolution
Framework

A 10-cycle smoke run across INTC, NVDA, AAPL, and COIN validated the hybrid genetic + LLM framework in 13 minutes with zero tokens

Genetic evolution searches hypothesis space for free — zero LLM tokens, millions of candidates evaluated. But the LLM excels at interpretation: understanding why a tree works, detecting regime patterns, and redirecting experiments toward productive regions. Guided Evolution gives each system the job it does best.

Stack$Trader Research TeamWHIPSAW  ·  March 2026
11 min read

The STAR agent (Systematic Trigger Allocation Research) and the veto evolution system revealed an asymmetry over the past two weeks: genetic evolution searches hypothesis space for free—zero LLM tokens, millions of candidates evaluated—while the LLM burns approximately 5,600 tokens per iteration doing the same population-based search. But the LLM excels at something the genetic algorithm cannot do at all: interpretation. Understanding why a tree works, detecting regime-level patterns, and redirecting structural experiments toward productive regions of the search space. Today I built Guided Evolution, a hybrid framework that gives each system the job it does best. The 10-cycle validation run completed in 13 minutes across 4 tickers, consumed 0 tokens, and surfaced NVDA’s strong preference for NOT() gate structures—a finding that would have taken the STAR agent 100+ iterations to hypothesize.

The Problem: Two Systems, One Search Space

The Stack$Trader veto evolution system, implemented in scripts/veto_evolver.py, breeds boolean gate trees through tournament selection, crossover, and mutation. Over 15 evolution passes, it has produced production-grade veto champions for 35 tickers with an average ΔSharpe (delta Sharpe ratio versus always-on) of +0.68. Each pass runs in roughly 40 minutes and costs nothing.

The STAR agent, separately, uses Claude Sonnet to propose entry hypotheses, interpret backtest results, and iterate. It produced strong results for INTC in 100 iterations but consumed approximately 560,000 tokens. The two systems search overlapping hypothesis space—they both attempt to find the conditions under which selling covered calls is optimal—but they attack the problem from opposite ends. Evolution is cheap and exhaustive but blind. The LLM is expensive but literate.

Guided Evolution merges them. The genetic algorithm does the heavy lifting: population-based search, many generations, zero cost. Sonnet checks in periodically—approximately 14,000 tokens per cycle—as a “research director” who interprets evolved trees and redirects the next cycle’s search.

Architecture

The framework runs a four-phase loop, implemented in scripts/star/guided_evolution.py (807 lines). Each cycle processes one ticker in round-robin order.

For each cycle: 1. EVOLVE (0 tokens) Genetic search via VetoEvolver Cycle 1–N: warm-start from production champions + variants Cycle N+: LLM-proposed seed trees from prior REDIRECT 2. INTERPRET (~9K tokens) Sonnet analyzes top-10 trees, indicator frequency, convergence trajectory, and train/val divergence 3. REDIRECT (~5K tokens) Sonnet proposes 3–5 seed trees, indicator priority changes, and structural experiments for the next cycle 4. LOG (0 tokens) Checkpoint state + append JSONL log entry

The total token budget for a full 40-cycle overnight run is approximately 560,000 tokens (40 × 14K), consuming just 11% of the 5 million token nightly budget. The remaining 4.24 million tokens are available for deeper analysis passes if the framework detects stagnation or interesting regime transitions.

Warm-Start vs. Random Initialization

The first pass through each ticker warm-starts from the production champion and its top-5 variant trees, loaded via load_champions() and load_variants_as_individuals() from scripts/veto_evolver.py. Subsequent passes for the same ticker use either LLM-proposed seed trees (if the LLM is enabled) or random initialization. This design lets the framework first establish whether it can improve on the production baseline, then explore genuinely novel structures.

LLM Integration

The LLM calls use the claude CLI with --model sonnet, --output-format text, and --no-session-persistence. Each call receives a structured system prompt (the INTERPRET_SYSTEM or REDIRECT_SYSTEM template) and a context payload including the top-10 evolved trees, indicator frequency counts, fitness trajectory samples, and any prior-cycle insights for the same ticker. The _parse_json() function handles markdown fence stripping and partial JSON extraction for robustness against formatting variation.

Finding: NVDA Favors NOT() Gates

All three NVDA cycles produced NOT() patterns—NOT(iv_rank_high), NOT(macd_accel), NOT(aroon_bullish)—with veto rates between 37% and 49%. NVDA benefits most from knowing when not to sell.

The most striking result from the smoke run was NVDA’s consistent convergence toward NOT() gate structures. Across three independent cycles—one warm-started from production, two randomly initialized—the genetic algorithm found that the optimal veto strategy for NVDA is to block covered call sales when specific indicators are not in favorable states. The best tree, NOT(iv_rank_high) at fitness 0.9448, vetoes when implied volatility rank is not elevated—sell calls only in high-IV environments, where premium is fattest relative to the risk of assignment.

This pattern makes intuitive sense for a high-beta semiconductor name. NVDA’s realized volatility clusters in regimes: calm drifts interspersed with sharp earnings moves, sector rotations, and AI-narrative catalysts. During quiet periods, covered call premium is thin relative to upside capture risk. The NOT() gate structure encodes precisely this logic: veto the call sale when the volatility environment does not justify the premium received.

The veto rates reinforce the finding. NVDA’s trees vetoed 37–49% of bars, far higher than AAPL (6.6–7.9%) or INTC (14.0–24.5%). For a volatile underlying, selectivity matters more than frequency—fewer, better-timed call sales outperform the always-on baseline by a wider margin.

Warm-Start vs. Random: Both Have Value

The validation run alternated between warm-started cycles (seeded from production champions) and randomly initialized cycles. The results showed that warm starts find strong trees faster—NVDA cycle 2 converged at generation 29 with fitness 0.9448—but random starts sometimes discover entirely different competitive structures. AAPL cycle 7 (random, vrp, fitness 0.2330) beat cycle 3 (warm, avwap_sl_near, fitness 0.1909). COIN cycle 8 (random, support_resist, fitness 0.3778) beat cycle 4 (warm, AND(aroon_bullish, earn_avoid), fitness 0.3562).

This is the expected behavior. Warm starts exploit the known-good region of the search space. Random starts explore globally. Both strategies contribute to the final champion set. The guided evolution framework’s round-robin cycling naturally alternates between them, and the LLM’s REDIRECT phase can further steer exploration toward under-explored structural families.

Convergence Behavior

The stagnation threshold of 15 generations without improvement triggered frequently, causing most cycles to terminate early at 19–30 generations out of the 30 maximum. With a population size of 60, the algorithm converges rapidly to a single dominant tree—indicator frequency analysis showed 10/10 dominance (100% of the top-10 trees sharing the same trigger) in every cycle. This is a known limitation of small populations: genetic diversity collapses early.

The full overnight configuration will use population 120 with 50 generations per cycle, which should maintain greater diversity through the mid-evolution plateau. The LLM’s REDIRECT phase can also inject structurally diverse seed trees to counteract premature convergence, something the pure genetic run cannot do.

Three-Way Comparison: A Data Limitation

The scripts/star/compare_star_veto.py tool ran STAR-only, Veto-only, and Combined backtests for all four tickers. All three modes produced identical validation alpha for INTC, NVDA, and AAPL. This is not a code bug—it reflects a data limitation. The validation window (approximately February 16 to April 1) contains too few daily bars for meaningful differentiation. With only 0–1 trades occurring during validation, and veto trees with low rates never firing in those windows, the three configurations are indistinguishable. Five-minute bars, which produce an order of magnitude more data points, would resolve this limitation.

Overnight Configuration

The framework is deployed via scripts/star/overnight_guided.sh, configured for nightly execution at 10pm via cron.

python3 scripts/star/guided_evolution.py \
    --tickers INTC,NVDA,AAPL,SPY,COIN \
    --cycles 40 \
    --generations 50 \
    --population 120 \
    --freq daily \
    --hours 8 \
    --token-budget 5000000

At these parameters, each cycle will run 1–2 minutes per ticker. The LLM’s INTERPRET and REDIRECT phases add approximately 30–60 seconds of latency per cycle (Sonnet response time plus JSON parsing). The 8-hour budget provides ample headroom for 40 cycles. Checkpoint/resume support (--resume) ensures robustness against interruptions: the framework saves state to scripts/star/_guided_checkpoint.json after every cycle and appends structured results to scripts/star/guided_evolution_log.jsonl.

One gap remains before the first full run: SPY is not in PORTFOLIO_SHARES and will be silently skipped. Adding SPY to the dictionary in scripts/veto_evolver.py is a one-line fix.

What Comes Next

The immediate next step is the first full overnight run with LLM guidance enabled. The comparison between the smoke run’s --skip-llm results and the LLM-guided results will reveal the marginal value of Sonnet’s interpretation and redirection—whether informed seed trees and structural experiments accelerate convergence beyond what pure genetic search achieves with the same compute budget.

If the LLM-guided run matches or exceeds production fitness for any ticker, the sync_production.py bridge will promote those champions into the live veto_champions.json file, completing the loop from overnight research to morning trade recommendations. The March 12 daily scan, generated at 07:25, already shows the production veto system in action: 24 tickers signaled SELL CC while 11 were vetoed, including COST, ARKK, and GLD.
