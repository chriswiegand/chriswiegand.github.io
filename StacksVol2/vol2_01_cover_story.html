How a genetic algorithm went from convergence collapse to a 35-ticker diversity breakthrough in 72 hours — and what Karpathy’s AutoResearch had to do with it

Over four versions and three sleepless nights, the AGENT system evolved from a textbook genetic algorithm into a hybrid framework that combines population-based search with LLM-guided interpretation. The trajectory mirrors Karpathy’s vision but applies it to a fundamentally different domain: not training language models, but evolving boolean gate trees for a 35-ticker covered-call portfolio.

Stack$Trader Research TeamWHIPSAW  ·  March 2026
22 min read

On March 5, 2026, Andrej Karpathy released AutoResearch — a minimalist framework where an AI agent autonomously designs, runs, and evaluates machine learning experiments in five-minute sprints on a single GPU. His README opened with a provocation: "One day, frontier AI research used to be done by meat computers... That era is long gone." Eight days later, Stack$Trader's AGENT system — Agent Guided Evolution Neural Trainer — has completed its fourth major version, evolving from a straightforward genetic algorithm into a hybrid framework that combines population-based search with LLM-guided interpretation. The trajectory mirrors Karpathy's vision but applies it to a fundamentally different domain: not training language models, but evolving boolean gate trees and exit parameters for a 35-ticker covered-call portfolio. This is the full story of AGENT's four generations, the failures that forced each redesign, and the roadmap for V5.

The Karpathy Inspiration

Andrej Karpathy's AutoResearch, released in early March 2026 on GitHub at karpathy/autoresearch, is deceptively simple. The system operates in what Karpathy calls the RSI loop — Recursive Self-Improvement. A human writes a program.md file defining the research goal. An AI agent (Claude 3.5 or GPT-4o) reads the instructions and modifies a train.py file. The system trains a small language model (from Karpathy's nanochat or nanoGPT family) for exactly five minutes. The agent evaluates the results using validation bits-per-byte (val_bpb). If the change improves performance, it becomes the new baseline. The agent then decides what to try next and repeats.

The genius is in the constraints. The repository contains very few files — prepare.py, train.py, and program.md — deliberately minimized so the LLM can understand and edit the codebase without breaking infrastructure. The agent works on a Git feature branch, creating commits for every successful improvement. It runs on a single consumer GPU, performing roughly 100 experiments overnight. The entire system operates as what Karpathy described as "a lab that works while you sleep."

Three properties of AutoResearch directly influenced AGENT's design:

  • Fixed time budgets per experiment. AutoResearch runs each training iteration for exactly five minutes. AGENT runs each evolution cycle for a fixed number of generations (30–50) with early stopping at stagnation. Neither system allows open-ended compute — the constraint forces efficiency.
  • Autonomous iteration with checkpointing. AutoResearch commits every improvement to Git. AGENT saves state to _guided_checkpoint.json after every cycle and appends structured results to guided_evolution_log.jsonl. Both systems are crash-safe and resumable.
  • The human shifts from practitioner to director. In AutoResearch, the human writes program.md and the agent handles the iteration loop. In AGENT, the human sets the ticker universe, parameter bounds, and fitness function, and the system handles the exploration. The role is managing agents, not writing training code.

But where AutoResearch uses a single LLM agent to both propose and evaluate changes to a training script, AGENT splits the workload between two fundamentally different optimization engines. The genetic algorithm does cheap, exhaustive, population-based search. The LLM (Claude Sonnet) provides expensive but literate interpretation. The result is a system that costs 89% less than a pure-LLM approach while exploring a combinatorially larger search space.

The AGENT Architecture

AGENT optimizes covered-call strategies for Stack$Trader, a portfolio system that sells options against equity positions across 35+ tickers. The core decision at each time bar is binary: should the engine sell a covered call, or should a "veto tree" block the trade? A veto tree is a boolean expression composed of market indicators — NOT(iv_rank_high), AND(vrp, obv_rising), OR(term_struct, support_resist) — that, when it fires, prevents the engine from selling premium into an unfavorable environment.

The search space is formidable: 27 trigger indicators, combinatorial tree structures up to depth 4, and 14 co-evolved bet parameters (target delta, DTE range, exit criteria, roll behavior, sizing). The fitness function is walk-forward out-of-sample Sharpe ratio, computed across 3 folds of daily bars. Manual tuning had plateaued. A systematic search was needed. AGENT was built to be that search.

V1 The Baseline: Convergence Collapse

AGENT V1 (scripts/veto_evolver.py, commit bf720a9) implemented a textbook genetic algorithm: tournament selection, subtree crossover, point mutation, elitism. The genome was the veto tree alone — bet parameters fixed at engine defaults (target_delta=0.25, dte_min=21, close_at_frac=0.25). Fitness was walk-forward OOS Sharpe ratio across 3 folds.

The initial results looked promising. Across 35 tickers with sufficient data, LMT emerged as the top champion at 3.71 fitness. The system independently discovered NOT(iv_rank_high) in 4 of the top 10 trees, validating the intuition that high implied volatility periods are poor for premium selling on volatile names. But underneath the headline number, the distribution told a different story.

The 11–10–10 split revealed the core failure. Top-tier tickers converged to dominant tree structures early and never explored alternatives. Bottom-tier tickers collapsed into local optima. Instrumenting the loop showed the mechanism: by generation 15, tournament selection was consistently choosing the same 2–3 tree topologies. Crossover between identical parents produces identical children. Mutation was too gentle — single-node swaps — to escape the basin. By generation 20, 80% of individuals shared the same leaf signature.

The genetic algorithm had become a hill-climber wearing a trench coat. Tournament selection was choosing the same 2–3 topologies by generation 15, crossover between identical parents was pointless, and mutation was too gentle to escape.

V1 portfolio mean fitness: approximately 1.2. Maximum fitness: 3.71 (LMT). The system worked, but it was leaving alpha on the table for the majority of tickers.

V2 Genome Expansion: Necessary but Insufficient

V2 expanded the genome from tree-only to tree-plus-parameters. Fourteen parameters joined the search space, organized into four groups:

  • Profit-taking: close_at_frac (0.15–0.60), exponential close, half-life fraction
  • Adverse-move stop: stop_loss_enabled, multiplier (1.5–4.0x), cooldown (5–40 bars)
  • Re-entry tuning: reentry_delta_shift (0.00–0.15) to adjust after stop-outs
  • Roll and assignment: roll-up threshold, assignment threshold

A whipsaw penalty was introduced: strategies with stop-loss rates above 15% received a fitness deduction. This prevented the optimizer from discovering trees that generated high Sharpe ratios by over-trading — entering and immediately stopping out in a pattern that happened to look profitable in-sample.

The result was real but marginal. The fitness distribution maintained the same 11–10–10 tiered split. Top-tier tickers saw modest improvement from optimized exits, but the convergence dynamics were unchanged. The algorithm was still collapsing populations to a single dominant tree by generation 20, burning 30+ generations of compute iterating on clones. Portfolio mean fitness: approximately 1.3. Maximum: 3.71 (LMT, unchanged).

The lesson was pointed: expanding the genome without fixing selection pressure wastes the expanded space. V2 added 14 parameters but the algorithm optimized them for the same trees it had already converged to, producing marginal improvements. The genome expansion was necessary — later versions would exploit it — but it was insufficient without diversity enforcement.

V3 The Diversity Breakthrough

V3 was the architectural pivot. Five interconnected mechanisms were introduced, each targeting a specific failure mode identified in V1 and V2.

1. Fitness Sharing by Leaf Signature

The centerpiece of V3. After each generation, individuals are grouped by leaf signature — the set of trigger indicators appearing in their tree, ignoring topology. Each individual's tournament-selection fitness is divided by the square root of its niche size: shared_fitness = raw_fitness / sqrt(niche_size). A tree topology shared by 9 individuals gets its selection fitness divided by 3. A unique topology keeps full fitness. This creates sustained pressure for novel structures even when a dominant topology has high raw fitness.

2. Clone Rejection in Elitism

V1/V2 carried the top N individuals forward unchanged. If the top 5 were all NOT(iv_rank_high), the elite pool was 5 identical trees. V3 scans a 3x wider pool and rejects duplicates by tree label, ensuring elite slots contain structurally distinct trees.

3. Adaptive Immigration

When stagnation exceeds a threshold, V3 escalates the fraction of the population replaced by random immigrants. At 5 generations of stagnation, 20% are replaced. At 10 generations, 33%. Increasingly aggressive diversity injection proportional to convergence severity.

4. Escalating Walk-Forward Efficiency Penalty

Walk-forward efficiency (WFE) — the consistency of OOS Sharpe across folds — is penalized with a weight that increases linearly through the run. Early generations are allowed high variance to explore; late generations must demonstrate consistency: effective_wfe = base_weight * (1.0 + gen / n_generations).

5. Calendar Artifact Penalty

V1/V2 allowed trees composed entirely of calendar triggers (monday, friday, opex_week) to compete on equal footing. These trees often exploited in-sample calendar effects. V3 applies a −0.30 fitness penalty to trees whose leaf set is a subset of calendar triggers.

New Search Space Dimensions

V3 also expanded the parameter bounds in two critical ways. The delta ceiling was raised from 0.55 to 0.70, allowing the algorithm to discover that high-beta names like COIN and PLTR benefit from selling closer to the money. And a new adaptive parameter, delta_iv_adjust (0.00–0.15), enabled the veto strategy to dynamically reduce target delta in high-IV environments: target_delta - delta_iv_adjust * (iv_rank / 100). This was the first parameter in the genome that changed behavior based on market state.

Cross-Ticker Meta-Learning

V3 introduced archetype-based cross-pollination. Tickers are classified as high_beta (COIN, TSLA, NVDA), stable (AAPL, MSFT, GOOG), or etf (VOO, VYM, GLD). The function _compute_meta_insights() analyzes which leaf indicators appear in high-fitness champions (above 1.5) across each archetype and computes win rates. High-win-rate indicators are more likely to seed the next cycle's initial population for related tickers.

This is the production system's "secret sauce" — cross-pollination of proven signals between related tickers, without requiring each ticker to independently discover the same pattern. GOOG's improvement from 2.28 to 3.25 came primarily through this mechanism: the stable-archetype analysis surfaced ma_uptrend and macro_positive as high-signal indicators from MSFT's champion, and seeded them into GOOG's population.

V4 The Hybrid: Guided Evolution

V3 solved convergence and established cross-ticker learning. V4 asked a different question: what if the search heuristic itself could reason?

This is where Karpathy's AutoResearch framework became the direct template. AutoResearch demonstrated that an LLM could autonomously manage the trial-and-error loop of ML research — proposing architectural changes, evaluating results, and iterating. AGENT V4 applied this principle to genetic evolution, creating a hybrid system where the genetic algorithm does the cheap, exhaustive population-based search and Claude Sonnet provides the expensive but literate interpretation.

The Architecture

V4's core loop, implemented in scripts/star/guided_evolution.py (807 lines), runs four phases per cycle:

  1. EVOLVE (0 tokens) — Genetic search via VetoEvolver. Cycle 1 warm-starts from production champions; subsequent cycles use LLM-proposed seed trees.
  2. INTERPRET (~9K tokens) — Sonnet analyzes top-10 trees, indicator frequency, convergence trajectory, and train/validation divergence.
  3. REDIRECT (~5K tokens) — Sonnet proposes 3–5 seed trees, indicator priority changes, and structural experiments for the next cycle.
  4. LOG (0 tokens) — Checkpoint state and append JSONL log entry.

The total token budget for a 40-cycle overnight run is approximately 560,000 tokens — 11% of the 5 million token nightly budget. Each cycle runs in 1–2 minutes per ticker. The system operates on a cron schedule via scripts/star/overnight_guided.sh, triggered at 10 PM nightly.

The Parallel Discovery: STAR

Concurrent with the guided evolution work, AGENT V4 incorporated the STAR framework (Systematic Tactical Alpha Research, scripts/star/star_agent.py). STAR used Claude as both Proposer and Critic in a Generative-Adversarial-style loop over the hypothesis space. The Proposer generates candidate entry conditions. The Critic evaluates them against backtest results and domain knowledge. 101 iterations on INTC 5-minute data completed overnight on March 11–12, consuming zero incremental API cost on the Max plan.

STAR's most important finding was that covered-call alpha is dominated by execution mechanics (dte_min=7, target_delta=0.25, max_positions=1) rather than entry timing. The top 10 hypotheses clustered in a $1,151 band ($7,756–$8,907 validation alpha) despite using fundamentally different indicator logic. The entry filter's value is in loss avoidance — vetoing catastrophic timing like selling into capitulation — not in identifying optimal moments. This insight directly informed how guided evolution weights tree fitness versus parameter fitness.

Each version addressed a specific failure of the previous version. V2 expanded the genome because V1 left exit parameters on the table. AutoResearch vs. AGENT: A Structural Comparison

Karpathy's AutoResearch

  • Domain: language model training
  • Search method: single LLM agent
  • Experiment unit: 5-minute training run
  • Metric: val_bpb (bits-per-byte)
  • Budget: ~100 experiments/night
  • Infrastructure: single GPU
  • Git workflow: feature branch, commit on improvement
  • Human role: write program.md

AGENT Guided Evolution

  • Domain: covered-call veto optimization
  • Search method: genetic algorithm + LLM advisor
  • Experiment unit: 50-gen evolution cycle
  • Metric: OOS Sharpe ratio (3-fold WFE)
  • Budget: 80 cycles in 2.54 hours
  • Infrastructure: single Mac CPU
  • Checkpoint: JSON state + JSONL log
  • Human role: set ticker universe + bounds

The fundamental difference is the division of labor. AutoResearch trusts the LLM to propose, execute, and evaluate — the entire research loop is LLM-driven. AGENT uses the LLM as an advisor to a fundamentally different search engine. The genetic algorithm explores the combinatorial tree space at zero token cost; Sonnet interprets the results and redirects the search at ~14K tokens per cycle. This hybrid approach is more efficient for domains where the search space is discrete and combinatorial (boolean trees) rather than continuous (neural architecture hyperparameters).

The convergence is in the philosophy. Both systems embody the same insight that Karpathy articulated: the repetitive parts of research — testing different configurations, evaluating results, deciding what to try next — can be automated. The human provides the fitness function, the constraints, and the domain knowledge. The system handles the iteration.

What Failed and Why It Mattered

The failures were as instructive as the successes. Each version's failure mode directly motivated the next version's architecture.

V1's failure: premature convergence. The population collapsed to a monoculture by generation 15. The fix was not to increase population size (a common but expensive remedy) but to change the selection dynamics. Fitness sharing attacked the root cause — identical individuals receiving identical selection pressure — rather than the symptom.

V2's failure: wasted genome expansion. Adding 14 parameters to a converged population produced 14 parameters optimized for the wrong tree. The lesson is general: expanding the search space is only useful if the search dynamics can exploit the expansion. V2 was necessary infrastructure for V3's breakthrough, but on its own it was nearly valueless.

V3's failure: blind diversity. V3 maintained population diversity and found better trees, but it had no mechanism to understand why a tree worked. It could not detect that NVDA's three independent cycles all converged to NOT() gate patterns, or that iv_rank_high was appearing across unrelated tickers, or that stable-archetype tickers shared structural preferences. It generated data but could not interpret it. V4's LLM layer was the answer.

V4's current limitation: production fitness gap. The guided evolution framework with 2 passes (max fitness 1.39) does not match production champions evolved over 15+ passes (max fitness 3.62). The gap is mechanical — 2-pass vs. 15+ pass is a 7.5x compute difference — but closing it requires either more passes or smarter initialization. The LLM-enabled overnight run, not yet completed, may close part of this gap by directing seed trees toward productive regions of the search space.

V5 Goals: Where AGENT Goes Next

V5 addresses six structural limitations in the current system. These are ordered by priority and dependency.

P0: Return-on-Capital Fitness

The current fitness function is Sharpe ratio, which is agnostic to capital efficiency. A strategy tying up $60,000 in AMZN shares for the same Sharpe as one using $4,500 in INTC shares is treated identically. V5 will introduce a return-on-capital component: fitness = sharpe * (premium_pnl / notional_deployed). This shifts the optimization toward strategies that maximize premium income per dollar deployed, directly aligning with portfolio-level capital allocation. Design is complete; implementation is a fitness function modification within the existing veto_evolver.py.

P0: Trade-Level Drill-Down

Current fitness evaluation produces a single aggregate number per ticker. V5 will decompose fitness into per-trade contributions. If a ticker's fitness is 1.5 but 3 of its 40 trades contributed −2.0 each, the system should surface those trades, their market conditions, and the tree's decision at that moment. This enables targeted evolution: the algorithm can focus mutation pressure on failure modes rather than evolving blindly. Combined with the LLM's interpretation layer, trade-level data gives Sonnet the granularity to make specific, actionable redirect decisions.

P1: Volume and Dollar Bars

AGENT currently operates on fixed daily bars, treating a 50-million-share day identically to a 5-million-share day. Volume bars (fixed share count per bar) and dollar bars (fixed dollar volume per bar) align with actual market activity, producing more stationary return distributions. The expected impact: more stable WFE across folds, particularly for variable-volume tickers like COIN and TSLA. This depends on the tick data pipeline, which is actively collecting data via the Schwab recorder (standalone_recorder.py).

P1: Fractional Overlay

The current engine treats coverage as binary: sell calls on the full position or skip. V5 will introduce fractional overlay — selling calls on a portion of the position and scaling up as the trade moves favorably. For a 500-share GOOG position, the optimizer might discover that selling 2 contracts initially and adding a 3rd at 40% profit-take produces higher risk-adjusted returns. This opens a new genome dimension: the scaling schedule.

P2: Sub-Daily Bar Resolution

Daily bars determine whether to sell; they cannot optimize when during the day to execute. V4 already supports 5-minute bars (scripts/intraday_veto_deep.py), and early results show fitness values averaging 2.9x higher than daily on the same tickers. V5 will fully integrate intraday optimization into the guided evolution loop, with the LLM interpreting cross-frequency patterns — for example, detecting that a ticker's daily veto tree and 5-minute entry timing tree use complementary indicators.

P2: Regime-Conditional Parameter Banks

V3's delta_iv_adjust is a single linear adjustment on one variable. V5 will generalize this to regime-conditional parameter banks: the genome will encode separate parameter sets for each HMM regime (bull, bear, sideways, high-vol), and the engine will switch between them at runtime. A ticker might evolve to sell at delta 0.40 with 15 DTE in bull regimes and delta 0.20 with 35 DTE in bear regimes. The parameter space grows 3–4x, but V3's diversity enforcement should prevent convergence collapse.

Beyond V5: Speculative Directions

Several research directions extend beyond the V5 roadmap. These are speculative but grounded in the patterns observed across V1–V4.

Multi-objective Pareto evolution. The current system optimizes a single scalar fitness. Real portfolio decisions involve trade-offs between Sharpe ratio, max drawdown, premium yield, and capital efficiency. A Pareto-frontier approach (NSGA-II or similar) would evolve a set of non-dominated solutions, letting the human choose the desired trade-off point rather than encoding it a priori in the fitness function.

Transfer learning between asset classes. The archetype-based cross-pollination in V3 groups tickers by qualitative similarity. A more rigorous approach would use learned embeddings — clustering tickers by their evolved champion tree structures and parameter distributions — to discover latent similarities that the archetype labels miss. If IBM and HON independently converge to vrp-based trees, the system should discover this structural kinship even though they are in different archetype groups.

Adversarial co-evolution. The current system evolves veto trees against historical data. An adversarial extension would co-evolve market scenarios alongside the trees: a "red team" generating synthetic bar sequences designed to exploit the champion tree's weaknesses, and the champion evolving to defend. This is the adversarial mode already prototyped in STAR (--adversarial flag) but not yet applied to the genetic evolution loop.

Full AutoResearch integration. The ultimate convergence with Karpathy's vision: a system where the LLM not only interprets and redirects but proposes structural changes to the evolution engine itself. V4's Sonnet can redirect which indicators to seed and which tree structures to try. A V6 Sonnet could propose changes to the crossover operator, the mutation rate schedule, or the fitness sharing formula — modifying the optimizer's code in the same way AutoResearch modifies train.py. The RSI loop applied not to a model but to the research tool that builds the model.

Lessons for Agentic Research

Three lessons from AGENT's development generalize beyond this specific system.

Hybrid search beats pure-LLM or pure-genetic. The genetic algorithm explored 210,000 tree evaluations across 35 tickers in 2.54 hours at zero token cost. No LLM-only system could match that throughput. But the genetic algorithm could not detect that 12/35 tickers converged to NOT() gates, or that iv_rank_high was the universal signal. The LLM sees patterns the GA cannot; the GA covers space the LLM cannot afford. The hybrid is strictly dominant.

Diversity enforcement is not optional. This is the lesson both Karpathy's work and AGENT independently discovered. AutoResearch's minimalist structure prevents the LLM from over-complicating the codebase — a form of complexity diversity enforcement. AGENT's fitness sharing prevents population homogeneity — a form of structural diversity enforcement. Without explicit pressure for novelty, any iterative optimization system converges prematurely. The specific mechanism matters less than the principle.

The human's job is to define the fitness landscape, not to traverse it. In AutoResearch, the human writes program.md. In AGENT, the human defines the ticker universe, parameter bounds, walk-forward fold structure, and fitness function. Neither system asks the human to propose specific experiments. The shift from "what should I try next" to "what does success look like" is the fundamental cognitive upgrade that agentic research provides. Karpathy called it the end of the era of "meat computers." In Stack$Trader's context, it is the end of manual hyperparameter tuning — and the beginning of a system that improves faster than I can supervise it.
