V13 Superpowers: Three Coupled Models for Multi-Timescale Trade Lifecycle

StackStrader's next architectural leap replaces the static entry-exit binary with a continuous adaptive system that classifies trade horizons, scales overlay dynamically, and evolves its own exit decisions at 5-minute resolution

Chris · March 17, 2026 · 14 min read

V12 fixed the exit-tree void and wired cross-asset features into the genetic algorithm. V13 asks a different question entirely: what if the system could learn that an INTC covered call sold at $1.94 and bought back 90 minutes later at $1.44 is a fundamentally different trade than holding to expiry — and discover both strategies on its own? The V13 “superpowers” plan introduces three coupled post-entry models — a Horizon Classifier, an Overlay Model, and an Exit Monitor — that together transform StackStrader from a system that makes binary sell/don’t-sell decisions into one that adaptively manages trade lifecycle across timescales from 30 minutes to 10+ days. The first two components, PositionState and ExitMonitor, are already implemented and tested in the new v13/ package (887a94d).

The Problem V13 Solves

Through V12, StackStrader’s trade lifecycle follows a rigid pattern. The genetic algorithm (GA) evolves veto trees that decide whether to sell a covered call on a given bar. If the tree says “sell,” the engine opens a position with fixed parameters — a static overlay percentage, a fixed profit target, and exit logic that operates at daily resolution. The system cannot distinguish between a 90-minute scalp opportunity and a 3-day carry trade. It uses the same delta, the same DTE selection, and the same profit target for both.

This limitation is not hypothetical. The design spec (docs/superpowers/specs/2026-03-16-v13-multi-timescale-trade-lifecycle-design.md) grounds the problem in a real trade: an INTC covered call sold at $1.94 and bought back at $1.44 ninety minutes later, capturing 26% of premium. The current system cannot recognize this as a scalp, cannot set a 20–35% profit target appropriate for that horizon, and cannot learn that INTC’s intraday behavior (strong fade tendency, mean-reverting) makes it an ideal scalp candidate while NVDA’s momentum profile calls for a different holding strategy entirely.

Four specific capabilities are missing from V11/V12:

  1. Recognizing that different setups call for different holding horizons
  2. Dynamically adjusting overlay based on IV regime and position evolution
  3. Opportunistically closing early when profit arrives faster than expected
  4. Learning per-ticker intraday behavior patterns

V13 addresses all four with a layered architecture that sits after the entry decision, managing everything that happens between opening a position and closing it.

Architecture: Three Models, One Shared State

The V13 design introduces three specialized models that operate at different frequencies, loosely coupled through a shared PositionState dataclass and co-evolved via a joint fitness function.

Entry signal (from any source) → "Enter"
    → Horizon Classifier picks trade class + initial params
    → Engine opens position with class-specific delta/DTE/overlay
    → Every 5 min: Overlay Model adjusts overlay %
    → Every 1-5 min: Exit Monitor decides hold/close/tighten/extend
    → Close → outcome feeds back to all three models' fitness

A critical design decision: these models are entry-source agnostic. They consume a minimal interface — (ticker, entry_bar_idx, entry_price) — and do not care whether the entry signal came from Pythia (the 20-agent Mixture-of-Experts ensemble in agents.py) or from the AGENT GA (the veto evolution system in veto_evolver.py). This preserves the independence of StackStrader’s two trading systems while allowing either to benefit from adaptive trade management.

Design Principle: Entry-Source Agnostic

Pythia and the AGENT GA are independent entry systems. V13’s three models manage the post-entry lifecycle only. The interface is (ticker, entry_bar_idx, entry_price) — nothing more. This keeps the systems composable: a future “best of both” entry system can plug in without any V13 changes.

The PositionState Protocol

All three models and the intraday engine share a common state object, already implemented in v13/position_state.py (887a94d). It captures everything a model needs to make a decision: entry snapshots (price, premium, IV rank, time), current state (price, IV, overlay percentage, contracts short), derived fields (P&L fraction, time fraction, P&L velocity, underlying percent change), and a V12 bridge field (v12_exit_signal) that passes the daily exit tree’s recommendation into the intraday decision loop.

@dataclass
class PositionState:
    ticker: str
    horizon_class: str                    # "scalp", "day_trade", "swing", "full_carry"
    horizon_probs: dict[str, float]       # Original classifier output

    entry_bar_idx: int
    entry_price: float
    entry_premium: float
    entry_iv_rank: float
    entry_time: datetime

    profit_target: float                  # GA-evolved per horizon class
    stop_loss: float
    overlay_base: float
    overlay_max: float

    # Derived (computed each bar by update_derived)
    pnl_frac: float       # P&L as fraction of profit target
    time_frac: float      # time elapsed as fraction of horizon
    pnl_velocity: float   # slope of recent P&L (30-min window)
    underlying_pct_change: float

    v12_exit_signal: bool  # Would V12 daily exit tree close today?

The engine creates this object at entry and updates its derived fields each bar via update_derived(). The P&L velocity is computed as the linear regression slope of the last 6 observations (30 minutes at 5-minute bars) — a simple but effective measure of whether profit is accelerating or stalling. Models read from the state and return actions; they never mutate it directly.

Component 1: Ticker Behavior Profiler

Before the three models can operate, they need per-ticker behavior data. The Ticker Behavior Profiler (v13/ticker_profiler.py, planned) is a nightly batch job that computes a rolling 20-day feature vector from 1-minute and 5-minute bars across all 41 tickers.

Ticker Profiler Feature Set
Feature What It Measures Method
Fade tendency Does the ticker give back morning gains? Autocorrelation at 15min/60min/session lags
Intraday vol shape When does volatility cluster? Vol profile curve in 30-min session buckets
Session return distribution How often does close oppose first-hour direction? Empirical distribution, rolling 20-day window
IV responsiveness How fast does premium decay after vol spikes? Decay half-life of IV rank after >2σ events
Regime sensitivity Does behavior change in risk-off vs risk-on? Correlation with SPY/VIX regime state
Source: docs/superpowers/specs/2026-03-16-v13-multi-timescale-trade-lifecycle-design.md

The output is adaptive — a rolling window, not a static cluster label. As a ticker’s behavior shifts through earnings seasons, sector rotation, or regime changes, the profile updates. This is the foundation the Horizon Classifier reads at entry time: INTC’s negative autocorrelation (mean-reverter) versus NVDA’s positive autocorrelation (momentum) is the signal that drives horizon classification.

Component 2: Horizon Classifier

The Horizon Classifier runs once at entry time and answers a single question: what kind of trade is this? It outputs a probability distribution over four holding classes.

Horizon Classes — Expected Duration and GA-Evolved Parameters
Class Horizon 5min Bars Profit Target Stop Overlay Base
scalp 30min – 2hr ~24 0.20–0.35 -0.10–0.20 0.15–0.30
day_trade 2hr – session close ~78 0.30–0.50 -0.15–0.25 0.20–0.40
swing 1 – 3 days ~234 0.40–0.60 -0.20–0.30 0.25–0.50
full_carry 3+ days to expiry ~780 0.70–0.90 -0.25–0.40 0.30–0.60
Parameter ranges are GA-evolved. Exact values are per-generation, per-individual.

The classifier is a supervised model (gradient-boosted trees or random forest) trained on simulated trade outcomes. For every potential entry bar in the training window, the pipeline simulates covered call opens across all four horizon classes, measures risk-adjusted outcome (premium_captured / max_drawdown), and labels that bar with the class that performed best. The model learns which features at entry predict which horizon class is optimal.

Critically, the classifier outputs probabilities, not a hard assignment. An output of {scalp: 0.6, day_trade: 0.3, swing: 0.1, full_carry: 0.0} tells the Exit Monitor “probably a scalp, but could stretch to a day trade.” If no class exceeds a GA-evolved minimum confidence threshold (initialized at 0.35), the system falls back to full_carry with default V12 parameters — the behavior the system already knows how to execute. The fallback rate is tracked as a diagnostic; high fallback rates signal insufficient training data or features.

Input features include both core features (always available: ticker profile, IV rank, IV term structure slope, time of day, session price action, relative volume, VIX, days to earnings) and optional Pythia features (HMM regime state, earnings catalyst score, sentiment composite) that improve accuracy when available but are not required. The classifier works with core features alone, making it usable regardless of which entry system triggered the trade.

Component 3: Overlay Model

Once a position is open, the Overlay Model runs every 5–15 minutes and adjusts the overlay percentage — how many contracts of the covered call are sold relative to shares held. Unlike the current system’s fixed overlay, V13’s overlay adapts to IV regime, position evolution, and time remaining.

The core scaling logic uses three GA-evolved parameters per horizon class:

overlay_target = overlay_base + iv_scale * max(0, iv_rank - iv_threshold)

# Additional modifiers (additive, then clamped):
raw = overlay_target
    + underlying_up_bonus * max(0, underlying_pct_change)
    + underlying_down_brake * min(0, underlying_pct_change)
    + time_decay_ramp * max(0, time_frac - 0.7)

final_overlay = clamp(raw, overlay_base, overlay_max)

The logic is transparent: iv_scale adds overlay when IV is rich (capturing richer premium), underlying_up_bonus adds overlay when the stock moves up (higher strikes become tradeable), underlying_down_brake reduces overlay when the stock drops (protecting against assignment risk), and time_decay_ramp reduces overlay as the horizon boundary approaches without the profit target being hit. All parameters are evolved by the GA, not hand-tuned.

For scalp-class trades, the 15-minute cooldown between adjustments means overlay effectively stays at its initial overlay_base for the entire trade. This is intentional — scalps are about capturing quick premium, not optimizing overlay. The Overlay Model adds the most value for day_trade and swing classes where positions persist long enough for IV regime changes to matter.

Component 4: Exit Monitor

The Exit Monitor is the most consequential of the three models. It runs every 1–5 minutes (depending on horizon class) and makes the continuous hold-or-close decision that V11/V12 could not express at intraday resolution.

The implementation in v13/exit_monitor.py (887a94d) is a GA-evolved binary decision tree — the same evolutionary framework that produces the entry veto trees in V12, extended to a richer action space. Each leaf node outputs one of four actions:

Exit Monitor Actions
Action Meaning Engine Effect
hold Trade is on track No action
close Take profit or stop loss Buy back calls, exit position
tighten Conditions shifting Reduce profit target or raise stop
extend Horizon prior was wrong Update horizon class upward (scalp → day_trade)
Source: v13/exit_monitor.py. Actions are GA-evolved leaf nodes in a binary decision tree.

The decision tree evaluates 9 real-time features: pnl_frac, time_frac, pnl_velocity, underlying_pct_change, iv_change, fade_tendency, spread_pct, time_of_day_frac, and horizon_confidence. A representative evolved tree might look like:

IF pnl_frac > 0.7 AND time_frac < 0.3:
    → close  (captured most of target way early — take it)
ELIF iv_velocity < -0.2 AND pnl_frac > 0.4:
    → close  (vol crushing, grab what's there)
ELIF time_frac > 0.8 AND pnl_frac < 0.15:
    → close  (running out of time, cut the loser)
ELIF fade_tendency > 0.5 AND underlying_up > 0.01 AND pnl_frac > 0.3:
    → close  (mean-reverter is extended, take profit before fade)
ELSE:
    → hold

The genetic operators — random_exit_tree(), mutate_exit_tree(), and crossover_exit_trees() — are fully implemented and tested. Mutation can change a leaf’s action, a branch’s feature, its comparison operator, or its threshold (via Gaussian perturbation with σ=0.1). Crossover swaps a random subtree between two parents. The ExitMonitor wrapper adds check-interval logic: it only evaluates the tree at configurable bar intervals, returning HOLD on non-check bars to avoid unnecessary computation.

Composition with V12 Exit Trees

V13’s Exit Monitor does not replace the V12 exit tree. The two systems compose based on horizon class:

  • full_carry trades: V12 exit tree is the primary exit mechanism. Daily resolution is appropriate for multi-day positions. The Exit Monitor is not invoked.
  • swing trades: Both systems contribute. The V12 tree provides a daily “should I still be in this?” check. The Exit Monitor adds intraday granularity. Either can trigger an exit (logical OR).
  • scalp and day_trade trades: The V12 tree is irrelevant (trade resolves within the day). The Exit Monitor is the sole exit mechanism.

The V12 exit tree’s daily recommendation is also available as an input feature to the Exit Monitor via the v12_exit_signal field on PositionState. This allows the intraday model to learn that “when the daily tree says close, pay attention” without being forced to obey it unconditionally.

The Intraday Backtesting Engine

The current engines (engine.py, engine_fast.py, engine_numba.py) operate on daily bars. They cannot simulate a 90-minute covered call scalp. V13 introduces v13/engine_intraday.py — a pure-Python engine that runs the full trade lifecycle at 5-minute resolution.

Architecture Decision: Pure Python + Vectorized Premiums

The existing Numba engines pre-compute signals before entering a JIT-compiled loop. V13’s overlay and exit decisions depend on evolving position state and cannot be pre-computed. The solution: the bar-by-bar loop is Python (so it can call model objects directly), but premium decay curves are pre-computed as numpy arrays using Black-Scholes + the IV surface from broker/chain_reconstructor.py, then indexed by bar. This avoids the Numba/Python boundary problem entirely.

The compute budget is substantial. 41 tickers at 78 bars per session over 40 trading days yields 127,920 bars per individual evaluation. With a population of 70 and 50 generations, that is 3,500 evaluations. At approximately 80 seconds per evaluation with 8-core multiprocessing (10 evaluations in parallel), a full evolution run takes roughly 7.8 hours. The plan includes a fallback: subsample to 10 representative tickers (stratified by behavioral archetype) for development iterations, reserving the full 41-ticker suite for final validation runs.

The interface returns an IntradayResult containing per-trade lifecycle data, horizon accuracy (fraction of trades that resolved in the predicted class), overlay efficiency (total premium captured per unit of overlay risk deployed), and exit quality (average realized P&L divided by optimal P&L across all trades).

Coupled Fitness: Co-Evolving Three Models

The most architecturally novel part of V13 is the coupled fitness function. All three models are evaluated together as a single genome, with the GA optimizing their joint performance.

Fitness = w1 * horizon_accuracy + w2 * overlay_efficiency + w3 * exit_quality
Coupled Fitness Components
Metric What It Measures Pressure Applied
horizon_accuracy Did the trade resolve in the predicted class? Classifier learns to predict actual outcomes
overlay_efficiency Premium captured per unit of overlay risk Overlay Model learns IV-aware scaling
exit_quality Realized profit vs. optimal exit Exit Monitor learns when early close beats holding
Weights w1, w2, w3 are candidates for a Bayesian sweep using the same skopt infrastructure as V12 Fix 5.

The exit quality metric deserves attention. It compares each trade’s realized P&L against the optimal exit — the best possible close within the horizon window, computed in hindsight. An Exit Monitor that closes at 80% of eventual max profit in 20% of the time/risk earns high fitness. One that closes early on a trade that would have doubled earns low fitness. This creates precise evolutionary pressure to learn when early exit genuinely beats holding — the hardest question in trade management.

The coupling between models is bidirectional. If the Horizon Classifier says “scalp” but the Exit Monitor consistently extends trades to day-trade length, the classifier’s horizon accuracy drops, penalizing its fitness. Over generations, the two models align. The same genome is evaluated across all 41 tickers globally — a scalp profile that works on INTC but fails on NVDA is penalized. The ticker behavior profile allows the system to apply different strategies to different tickers within a single genome, encoding rules like “if fade_tendency > 0.5 and IV_rank > 60, then scalp” rather than hard-coding ticker assignments.

Relationship to V12 Fitness

The V12 fitness function (growth/drawdown/Sharpe/capital-efficiency on daily equity curves) continues to evaluate entry gate quality. The V13 coupled fitness evaluates trade lifecycle quality on intraday equity curves. These are separate evaluations applied to separate genomes. The V12 genome encodes which bars to enter on; the V13 genome encodes how entries are managed. The two evolve independently but are evaluated together at the portfolio level: entry rules that produce entries the V13 models cannot manage well see lower portfolio Sharpe, creating indirect selection pressure for coherence between entry and lifecycle.

Staged Evolution: Managing Complexity

Co-evolving three coupled models introduces complex dynamics. The design spec identifies the convergence risk directly and prescribes a staged approach:

  1. Horizon Classifier alone — 20 generations with fixed overlay and fixed exit rules. Establish a baseline for horizon accuracy.
  2. Add Exit Monitor — 20 generations with evolved exit trees and fixed overlay. Establish a baseline for exit quality.
  3. Add Overlay Model — 20 generations with all three models active. Establish a baseline for overlay efficiency.
  4. Full coupling — 50+ generations with joint fitness. If coupled fitness stalls for 10 consecutive generations, fall back to independent fitness evaluation.

This staged approach also serves as a diagnostic. If horizon accuracy plateaus in stage 1 at an unacceptable level, the classifier needs more features or training data before proceeding. If exit quality is high in stage 2 but collapses when overlay is introduced in stage 3, the overlay model is interfering with exit decisions — a signal to tighten the overlay cooldown constraints.

“Real trading requires continuous adaptive decision-making across multiple timescales. A covered call on INTC sold at $1.94 and bought back at $1.44 ninety minutes later — capturing 26% of premium — is a fundamentally different trade than holding to expiry. The system needs to discover and execute both.”

What V13 Does Not Change

Equally important is what remains untouched. The GA infrastructure in veto_evolver.py continues as-is. The Numba engine for daily-resolution entry performance backtesting is not replaced. The AutoResearch pipeline still discovers new triggers. Pythia’s 20-agent ensemble remains independent. Gemini’s V13 pillars (ReAct agent, async workers, macro ingestion) are additive and proceed on their own schedule.

Execution Order and Data Requirements

The implementation follows a strict dependency chain: Ticker Profiler first (all models read from it), then the intraday backtester (cannot train models without simulation), then Horizon Classifier (entry-time decisions before in-trade decisions), then Exit Monitor (needs horizon prior), then Overlay Model (needs exit monitor to measure efficiency), and finally the coupled fitness function that ties them together. Gemini’s pillars (ReAct agent, async workers, macro ingestion) layer on top after the core lifecycle models are validated.

Data requirements are mostly satisfied. The system has 41 tickers with 1-minute through daily bars covering roughly two months (January 15 to March 16, 2026). Intraday options chain snapshots for realistic premium decay modeling are partially available in data/ticks/snapshots/. The gap is filled by Black-Scholes modeling with IV surface interpolation from broker/chain_reconstructor.py. A future validation dataset — the user’s 70 real brokerage trades — will be added when structured.

What V13 Means for the System

V13 represents the largest architectural expansion since the original genetic algorithm was introduced. Versions 1 through 12 operated on a single axis: should we sell a covered call on this bar, yes or no? V13 adds two new axes — what kind of trade and how to manage it — and does so at a temporal resolution 78 times finer than the daily bar loop. The total parameter space under GA control expands from entry gate trees to entry gates plus horizon class parameters plus overlay parameters plus exit trees, per horizon class, per ticker behavioral archetype.

The design is deliberate about what it does not attempt. It does not try to improve entry signals. It does not replace the existing daily-resolution backtesting infrastructure. It does not force Pythia and the GA into a single system. Each of those boundaries is load-bearing: the entry systems have demonstrated $75K of OOS alpha; the daily backtester validates entry gate quality with Numba-speed; the dual-system architecture allows independent experimentation. V13 layers on top without disturbing what already works.

The INTC scalp that opened this article — sold at $1.94, bought back at $1.44 in ninety minutes — is the simplest possible demonstration of what V13 enables. Today, the system would hold that position to expiry, hoping for full premium capture. V13’s Horizon Classifier would recognize INTC’s fade tendency and high IV rank as scalp conditions, the Exit Monitor would see P&L reaching 26% of premium in 18 bars and fire a close action, and the system would capture that 26% and free the capital for the next trade. That is the difference between a system that makes a single decision and one that manages a trade.
