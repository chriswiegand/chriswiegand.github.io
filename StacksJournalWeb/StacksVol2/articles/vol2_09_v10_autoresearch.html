Inside the v10 AutoResearch Loop

How six LLM-driven agents close the loop between hypothesis, synthesis, testing, and production in a covered-call veto evolution system

The v10 AutoResearch routine is the system that writes its own research. It mines failures from forward tests, synthesizes new trigger code via Claude, validates that code in a sandbox, and feeds confirmed improvements back into the genetic algorithm — all without human intervention.

ChrisEngineer  ·  March 15, 2026
18 min read

The v10 AutoResearch routine is the system that writes its own research. It mines failures from forward tests, synthesizes new trigger code via Claude, validates that code in a sandbox, A/B tests it against champion trees, and feeds confirmed improvements back into the genetic algorithm (GA) — all without human intervention. It runs as a sidecar to the main evolution loop, activated every 10 GA cycles or whenever stagnation exceeds 50% of tickers. In its first live run, it proposed 8 new triggers, 4 of which passed validation and entered the evolution vocabulary. This article dissects the architecture, module by module.

I The AutoResearch Vision

The inspiration is Karpathy's notion of "software 2.0" — systems where the model writes the code. In StackStrader, the GA evolves boolean veto trees that decide when not to sell covered calls. But the GA can only combine existing triggers; it cannot invent new ones. AutoResearch closes that gap. An LLM examines why the GA stalls, proposes new boolean trigger functions as executable Python, validates them against real market data, and injects the survivors into the next evolution cycle.

The key design constraint: the loop must be fully autonomous. No human reviews trigger code before it enters the GA. Safety comes from AST inspection, restricted namespaces, and sample-data execution — not manual review. The system trusts code that passes three gates: syntax safety, sandboxed execution on 5,000+ bars of SPY data, and a veto-rate sanity check (5%–95%).

II The 6-Phase Orchestrator

The orchestrator (scripts/autoresearch/orchestrator.py) coordinates six research phases that run sequentially within a single research cycle. Each phase receives the same context — champion trees, stagnation counts, latest GA results, and a research memory summary — and returns structured JSON directives.

The orchestrator's decision logic for when to fire is deliberately conservative:

def should_run_research(self, cycle_num, ticker_stagnation):
    # Guard: don't run back-to-back
    if cycle_num - self._last_research_at_ga_cycle < 3:
        return False

    # Periodic trigger: every 10 GA cycles
    if cycle_num > 0 and cycle_num % self.research_interval == 0:
        return True

    # Stagnation trigger: >50% of tickers stagnating >3 cycles
    if ticker_stagnation:
        total = len(ticker_stagnation)
        stagnating = sum(1 for s in ticker_stagnation.values() if s > 3)
        if stagnating / total > self.stagnation_threshold:
            return True

    return False

After all six phases complete, results are merged via _merge_phase_results(), which deduplicates triggers by name, deduplicates seed trees, and deep-merges fitness adjustments (later phases override earlier ones). The merged directives are written to research_directives.json, which the GA consumes on its next cycle. After consumption, the file is renamed with a .consumed_ timestamp suffix to prevent re-processing.

III Phase 1: Failure Mining

The FailureMiner (scripts/autoresearch/failure_mining.py) is the adversarial engine. It selects the worst-performing tickers from the checkpoint, reruns their champion veto trees on 2026 out-of-sample data, extracts the N worst trades by realized PnL, and asks the LLM to explain what went wrong.

Workflow

  1. Select the 5 lowest-fitness tickers from the champion checkpoint
  2. For each ticker, run the champion tree on forward data (post-train_cutoff)
  3. Pair OPEN/CLOSE trade events and sort by PnL ascending
  4. Enrich the 10 worst trades with indicator context (30 indicators at entry, 5-bar price action before/after)
  5. Call LLM (Sonnet) per ticker: "What pattern do these failures share?"
  6. Call LLM (Sonnet) cross-ticker: "What systematic blind spots exist?"

The indicator context capture is detailed enough to give the LLM real signal. For each bad trade, the miner extracts every computed indicator value at the entry bar:

def _get_indicator_context(df, trade_date, window=5):
    idx_loc = df.index.searchsorted(pd.Timestamp(trade_date))
    row = df.iloc[idx_loc]

    indicators = {}
    for col in df.columns:
        if col.lower() in {"open", "high", "low", "close", "volume"}:
            continue
        val = row.get(col)
        if val is not None and not (isinstance(val, float) and np.isnan(val)):
            indicators[col] = round(float(val), 6)

    return {
        "indicators_at_entry": indicators,
        "price_at_entry": float(df["close"].iloc[idx_loc]),
        "price_action_before": df["close"].iloc[max(0,idx_loc-5):idx_loc].tolist(),
        "price_action_after": df["close"].iloc[idx_loc+1:idx_loc+6].tolist(),
    }

The cross-ticker synthesis step is critical. Individual per-ticker analyses might identify idiosyncratic patterns, but the cross-ticker LLM call identifies systemic blind spots — failure modes that appear across AAPL, NVDA, and INTC simultaneously. These become the highest-priority inputs for trigger synthesis.

IV Phase 2: Trigger Synthesis

The TriggerSynthesizer (scripts/autoresearch/trigger_synthesis.py) is where hypotheses become executable code. It takes the failure patterns from Phase 1 and the existing trigger vocabulary (~84 triggers across 15 categories) and asks Claude Opus to write new Python functions.

VI Phase 4: Fitness Function Review

The FitnessReviewer audits whether the fitness function is producing good champions or enabling gaming. It checks for four failure modes: high Sharpe on few trades, excessive veto rates, concentrated profit, and calendar-only trees.

VII Phase 5: Literature Scan

The LiteratureScanner uses academic finance knowledge to propose structurally-motivated signals grounded in options pricing theory, market microstructure, and behavioral finance (e.g., Garleanu & Pedersen 2011, Bollen & Whaley 2004).

VIII The Closed Loop

Phase 6 is where the loop closes. Untested synthesized triggers from Phase 2 are compiled, registered into the live TRIGGER_MODES dictionary, and A/B tested against baseline fitness. If a trigger improves fitness, it is marked as confirmed in the research memory.
