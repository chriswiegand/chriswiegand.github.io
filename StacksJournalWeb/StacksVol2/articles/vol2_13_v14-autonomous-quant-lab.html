V14: Autonomous Quant Lab Overview

An API layer that turns the StackStrader engine into a JSON-callable research instrument for LLM orchestrators, with dynamic code injection, semantic telemetry, and a sandboxed experiment pipeline.

Chris · March 17, 2026 · 11 min read

V14 is not a new engine. It is not a new set of indicators. It is a 688-line API layer (v14/) that wraps the existing V12 engine and exposes it as a set of JSON-callable tools for an LLM orchestrator — transforming StackStrader from a system that an AI advises into one that an AI directly operates. Where V10–V13 used the genetic algorithm (GA) as the primary research driver, with LLMs proposing mutations and reading logs on the side, V14 removes the GA entirely for exploratory research. The orchestrator formulates a hypothesis, writes a custom AlphaModel as Python code, injects it into the engine at runtime, runs a single backtest, reads the semantic telemetry report, and decides what to try next — all in a single reasoning chain.

V14 at a Glance

  • 4 modules, 688 total lines of production code
  • 5 commands: backtest, benchmark, analyze, cleanup, ping
  • ~5–15 seconds per single experiment (vs. 70–110 sec/generation in V13)
  • Full Python injection: arbitrary AlphaModel classes compiled and loaded at runtime
  • Isolated sandbox: all experimental code lives in v14/sandbox/, separate from production V12/V13

File System Structure

The v14/ directory is isolated from the production V12/V13 codebase by design. No V14 code is imported by the engine, the GA launcher, or the autoresearch pipeline. This isolation ensures that recursive synthesis — an AI writing and executing its own code — cannot corrupt the production trading system, even if a generated AlphaModel throws an exception or produces nonsensical signals.

The Brain Interface — orchestrator_api.py

The entry point for all AI-driven research. The V14Orchestrator class (237 lines) accepts a JSON command string via CLI, routes it to the appropriate method, and returns a JSON response. It handles three responsibilities that seem trivial but are essential for reliable orchestration:

  • Date alignment: Converts all timestamps to timezone-naive pd.Timestamp objects, preventing the silent NaN propagation that occurs when mixing tz-aware and tz-naive indices.
  • VIX joining: Automatically joins daily VIX data to intraday bars via date-key mapping. The orchestrator never needs to know that VIX is daily while ticker data is 5-minute — the API handles the frequency mismatch internally.
  • Custom logic injection: When a command includes a custom_logic field (a Python code string and class name), the orchestrator delegates to the Dynamic Loader, which compiles the code and returns an instantiated object. This object is passed to V12's run_single() via the existing alpha protocol.

The Analyst — semantic_telemetry.py

The feedback translator. The SemanticAnalyzer class (286 lines) takes raw backtest results — a metrics dictionary and a trade log — and produces a structured narrative report that an LLM can reason about directly. This is the critical insight behind V14's design: an LLM can read "losses correlated with overbought entries (avg RSI 68.2)" and immediately formulate a follow-up experiment, but it cannot usefully reason about a flat list of 847 trade events.

The analyzer performs four operations:

  1. Failure mining: Clusters losing trades by exit event type (ASSIGN, STOP_LOSS, EXPIRE_ITM, CLOSE_DTE_FLOOR), then correlates with VIX level and RSI at entry. If average VIX on losing trades exceeds 25, the narrative recommends a VIX-cap veto. If average RSI exceeds 65, it suggests lowering the rsi_max threshold.
  2. Regime analysis: Maps each closed trade to the HMM regime state of its entry bar. Computes per-regime trade count, win rate, average PnL, and total PnL. This enables the orchestrator to ask: "What if I only trade during regime 2?"
  3. Missed opportunity detection: Currently a Phase 2 placeholder. Will require engine-level candidate logging to record bars where entry scored high but was vetoed.
  4. Narrative synthesis: Generates a tiered text report with performance status (exceptional/strong/neutral/negative based on Sharpe thresholds), critical failure patterns, regime breakdown, and strategic recommendations.

The Compiler — dynamic_loader.py

The most consequential 69 lines in the system. DynamicLoader.load_from_string() accepts a Python code string and a class name, writes the code to a UUID-namespaced file in v14/sandbox/ (e.g., v14_sandbox_a1b5bc1d.py), uses importlib.util.spec_from_file_location to compile it into a module, and returns an instantiated class. The caller — the orchestrator — passes this object to the engine, which calls its should_open(), score(), and should_close_early() methods during the backtest loop.

Safety Note: The current loader has no AST-level code restrictions. Before production use, V14 Phase 2 will add validation to prevent file I/O, network calls, and process spawning in injected code. During the current prototype phase, V14 runs only in local research sessions.

The Laboratory — v14/sandbox/

A temporary directory where each experiment's Python code lands as a uniquely named file. Seven sandbox files were generated during initial prototype testing, ranging from 300-byte pass-through alphas to 817-byte HorizonBiasAlpha classes with RSI-based horizon classification. The cleanup command removes all v14_sandbox_*.py files, resetting the laboratory for the next research session.

The Taskmaster — run_research_batch.py

A template for running multiple experiments in a single pass. The current prototype contains two experiments: a Horizon-Based Delta/DTE Bias test against AAPL (injecting a HorizonBiasAlpha class that classifies bars by RSI into scalp/swing/full_carry horizons) and a Co-Evolved Exit Tree test against NVDA (injecting an ExitTreeAlpha that triggers early close when RSI < 40 and MACD histogram is negative). Both benchmark against the default parameter baseline over 2025-06 to 2026-02.

The V14 CLI / MCP Specification

Every V14 interaction follows a single pattern: a JSON object with a command field and an args object, passed as a string argument to the orchestrator script.

python3 v14/orchestrator_api.py '{"command": "...", "args": {...}}'

The API returns a JSON string with a status field ("success" or "error") and command-specific payload fields. All numeric values are cleaned of NumPy types and NaN values before serialization.

V14 Command Reference
Command Purpose Required Args Returns
backtest Run a single backtest with optional custom logic ticker, params, start, end Sharpe, return, drawdown, trade count
benchmark Compare baseline vs. experiment params ticker, base_params, experiment_params, start, end Sharpe delta, both metric sets, improvement flag
analyze Run backtest + semantic telemetry report ticker, params, start, end Full narrative report with failure clusters
cleanup Remove all sandbox files none Confirmation message
ping Health check none "pong" + timestamp
Source: v14/orchestrator_api.py · handle_request()

The backtest and benchmark commands accept an optional custom_logic field in their args. This is the mechanism for recursive logic injection — the orchestrator writes Python code, embeds it in the JSON command, and the engine executes it during the backtest. The custom_logic object has two fields: code (the full Python source as a string) and class_name (the class to instantiate).

Example: Benchmark with Custom Logic Injection

The following command benchmarks a custom exit tree against NVDA's default parameters. The orchestrator hypothesizes that closing early when RSI drops below 40 during a negative MACD histogram will reduce drawdowns. The benchmark command runs both the baseline and experiment, then returns the Sharpe delta.

python3 v14/orchestrator_api.py '{
  "command": "benchmark",
  "args": {
    "ticker": "NVDA",
    "base_params": {},
    "experiment_params": {},
    "start": "2025-06-01",
    "end": "2026-02-01",
    "custom_logic": {
      "class_name": "ExitTreeAlpha",
      "code": "class ExitTreeAlpha:\n    def prepare(self, df, params): pass\n    def should_open(self, row, price, params, context=None): return True\n    def score(self, row, price, params, context=None): return 1.0\n    def should_close_early(self, row, current_price, params):\n        return row[\"rsi\"] < 40 and row[\"macd_histo\"] < 0"
    }
  }
}'

The response includes both metric sets and the critical sharpe_delta and improvement fields:

{
  "status": "success",
  "ticker": "NVDA",
  "sharpe_delta": 0.0342,
  "baseline": {"sharpe": 1.24, "ann_return": 0.18, "max_drawdown": 0.09},
  "experiment": {"sharpe": 1.27, "ann_return": 0.19, "max_drawdown": 0.07},
  "improvement": true
}

The AlphaModel Interface Contract

Any class injected via custom_logic must implement four methods. The engine calls them during the backtest loop at the appropriate lifecycle points:

class YourAlphaModel:
    def prepare(self, df, params):
        """Called once before backtesting. Enrich the DataFrame."""
        pass

    def should_open(self, row, price, params, context=None):
        """Called on each bar. Return True to allow entry."""
        return True

    def score(self, row, price, params, context=None):
        """Called on each bar. Return a float signal strength."""
        return 1.0

    def should_close_early(self, row, current_price, params):
        """Called each bar while a position is open. Return True to exit."""
        return False

System Architecture

V14 sits between the LLM orchestrator and the V12 engine. The orchestrator never touches V12 directly — all interaction flows through the API layer, which handles data loading, indicator computation, custom logic compilation, and result translation.

V14 Data Flow — From Hypothesis to Insight
LLM Orchestrator Hypothesis + Python Code
↓ JSON command via CLI
orchestrator_api.py "Brain Interface" · Route & Load
↙   ↘
dynamic_loader.py "Compiler" · Code → Object
V12 engine.run_single() Backtest Engine · alpha Protocol
↓ result + trade_log
semantic_telemetry.py "Analyst" · Metrics → Narrative
↓ JSON response
LLM Orchestrator Interpret → Next Hypothesis

The key architectural decision: V14 uses the alpha protocol that V12's run_single() already supports, rather than V13's hook system. This means V14 experiments are completely self-contained. The orchestrator writes a Python class, the Dynamic Loader compiles it, the engine calls its methods during the backtest loop, and Semantic Telemetry interprets the results. No genome, no population, no GA.

V10–V13 asked: "Can an AI improve a genetic algorithm?" V14 asks: "Can an AI replace the genetic algorithm entirely for hypothesis-driven research?"

How to Launch for Claude Collaboration

V14 is designed for an LLM orchestrator — specifically Claude — to autonomously drive quantitative research against the StackStrader engine. The collaboration workflow has five phases, executed in sequence within a single session or across multiple sessions synchronized via git.

Phase 1: Read the Guide

The orchestrator reads V14_GUIDE.md at the repository root to internalize the system's capabilities, the AlphaModel interface contract, and the available commands. This file is the source of truth for what V14 can do.

# Claude reads the V14 guide
Read: V14_GUIDE.md

Phase 2: Monitor V13 Artifacts

Before designing experiments, Claude reads the latest V13 evolution artifacts to understand the current state of the champion genome and where the GA is struggling:

  • artifacts/v13_champions.json — Current best genome, including exit tree structure, horizon biases, and overlay thresholds
  • artifacts/v13_diagnostics.json — Ablation results showing which V13 hooks help vs. hurt, plus cross-ticker contribution analysis
  • artifacts/veto_evo_report.md — Latest veto evolution status across tickers

This context allows the orchestrator to formulate hypotheses that target known weak points. For example, if ablation shows the overlay hook degrading fitness on COIN and IBIT, the orchestrator can design a targeted experiment to find overlay scaling rules that work for crypto-correlated tickers.

Phase 3: Command the Engine

The orchestrator issues V14 commands to test hypotheses. A typical research cycle:

  1. Analyze: Run analyze on a target ticker to get the semantic telemetry baseline. Read the failure clusters and regime breakdown.
  2. Hypothesize: Based on the narrative ("losses correlated with overbought entries, avg RSI 68.2"), formulate a specific fix ("lower RSI entry threshold to 60").
  3. Synthesize: Write a Python AlphaModel class that implements the hypothesis.
  4. Benchmark: Run benchmark with the custom logic against the baseline. Read the sharpe_delta.
  5. Iterate or deploy: If improvement: true, consider deploying the logic. If not, refine the hypothesis based on the experiment's own telemetry.
# Step 1: Get the baseline narrative
python3 v14/orchestrator_api.py '{
  "command": "analyze",
  "args": {
    "ticker": "AAPL",
    "params": {},
    "start": "2025-06-01",
    "end": "2026-02-01"
  }
}'

# Step 2: Benchmark a hypothesis
python3 v14/orchestrator_api.py '{
  "command": "benchmark",
  "args": {
    "ticker": "AAPL",
    "base_params": {},
    "experiment_params": {"rsi_max": 60},
    "start": "2025-06-01",
    "end": "2026-02-01"
  }
}'

Phase 4: Iterate on Results

The orchestrator reads each benchmark result, extracts the Sharpe delta, and decides the next action. Positive deltas above a significance threshold (typically +0.05 Sharpe) are worth pursuing. The orchestrator can chain experiments: test a parameter change, then layer a custom AlphaModel on top, then test across multiple tickers using the batch runner pattern.

Phase 5: Synchronize via Git

Successful experiments are persisted through git. The orchestrator saves promising AlphaModel code to a named file (not a sandbox UUID), documents the hypothesis and results in the journal, and commits. This creates a searchable record of what was tried, what worked, and why — which future orchestrator sessions can read to avoid redundant experiments.

Design Principle: V14 never modifies production code. All experiments run through the alpha protocol and sandbox. Promotion from V14 prototype to V13/V12 production is always a manual human decision — the AI proposes, the human disposes.

Limitations and Next Steps

V14 is a Phase 1 prototype. Three capabilities are explicitly deferred:

  1. Missed opportunity mining (_mine_missed_opportunities() returns a placeholder). Requires engine-level candidate logging to record bars where entry conditions scored high but were vetoed. This would enable the orchestrator to ask: "What if I removed this veto?"
  2. Code safety sandbox. The DynamicLoader currently executes arbitrary Python with no restrictions. Before any production or unattended use, AST-level validation must prevent file I/O, network calls, subprocess invocations, and sys.exit() in injected code.
  3. V14-V13 bridge. The orchestrator should be able to load V13's champion genome, translate its horizon/overlay/exit parameters into a custom AlphaModel class, and iterate on it with targeted experiments rather than blind evolution. This creates a feedback loop between V13's breadth (population-based search) and V14's depth (targeted structural experiments).

The 688 lines of V14 code were written during the same session that launched V13's 6-batch evolution across 38 tickers. The timing is not coincidental: V14 exists because V13's ablation results at gen 5 showed all three hooks hurting performance. Rather than wait 50 generations to see if the GA can recover, V14 gives the orchestrator the tools to investigate why — and to test structural alternatives that the GA's parameter-space mutations cannot reach.
