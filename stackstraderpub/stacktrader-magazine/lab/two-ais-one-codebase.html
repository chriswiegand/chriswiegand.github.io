Two AIs, One Codebase

Claude handles architecture and implementation. Gemini runs research and validation. Here is how two large language models divide labour on a real trading system — what works, what doesn't, and what it reveals about the future of AI collaboration.

By Stack$Trader Research Team  ·  March 2026  ·  8 min read

Stack$Trader is unusual in another dimension beyond its trading methodology: it is developed by two AI systems working in parallel, with clearly delineated responsibilities and a formal handoff protocol. This is not a marketing claim. It is a practical response to a real constraint: building a trading system from scratch requires both deep quantitative research capability and rigorous software engineering — and the combination is difficult to sustain in any single agent working in long, complex context windows.

The division of labour

Claude Implementation Lead
  • All production code — agents, engine, broker integration
  • Test suite design and maintenance (119 tests)
  • Architecture design documents for live deployment
  • Configuration management (supervisor_config.json)
  • Code review and integration of research findings
  • Performance engineering and optimisation
  • Continuous integration and deployment pipeline
  • Agent weight integration after Gemini optimisation
  • Implementation feasibility assessment
Gemini Research Lead
  • Academic literature review (options pricing, VRP, regime detection)
  • Statistical validation of all major findings
  • Backtest execution and analysis across ticker universe
  • Feature lift scans and agent weight optimisation
  • Alternative data source evaluation and cost analysis
  • Earnings cluster stress testing
  • Monte Carlo and walk-forward validation
  • Identification of new alpha opportunities
  • Research-to-implementation feasibility guidance

The split follows a principle that any research organisation would recognise: separate the people who discover things from the people who build things, but give them a tight collaboration protocol so nothing is lost in translation. In Stack$Trader's case, both "people" are AI systems — but the principle holds.

The handoff protocol

The coordination mechanism is HANDOFF.md — a living document in the repository that both agents read at the start of every session. It contains: the status of all active tasks, explicitly assigned to one agent; the queue of ready-to-work tasks; any blockers; the last three git commits; and the next synchronisation checkpoint.

The rule is simple: before starting any work, read HANDOFF.md. Before finishing any work, update HANDOFF.md. This prevents the failure mode where both agents attempt the same work, or where a handoff falls through because the receiving agent didn't know it was ready to receive.

Git commit messages serve as the secondary communication channel. A commit message like "Phase 6 Task 1A: Validate edge-in sampling strategy — RBF proof confirmed, ready for Claude implementation" is simultaneously a code record and a handoff signal. The commit history is, in effect, the project's audit trail.

The decision gate: research → implementation

"Before Claude builds anything: is this signal actually alpha, or just overfitting? Can we implement it cleanly? Will the benefit exceed the complexity cost?"

Every feature that moves from research to implementation passes through a three-question gate:

Gemini asks: Is this signal actually alpha, or am I observing a data-specific pattern that won't generalise? (Statistical rigour gate.)

Claude asks: Can this be implemented cleanly within the existing architecture without creating technical debt? (Engineering feasibility gate.)

Both ask: Will the expected benefit to ensemble Sharpe exceed the complexity cost of adding another component? (Marginal value gate.)

A feature that fails any of these three questions doesn't get built. This gate has already prevented several development paths that were statistically interesting but practically unimplementable (real-time options order flow) or whose marginal benefit didn't justify the integration complexity (proprietary volatility surface data at institutional pricing).

What works: asynchronous parallelism

The most valuable structural feature of the collaboration is that Claude and Gemini work asynchronously. While Claude is implementing Phase 6 Task 1B (edge-in sampling code), Gemini is simultaneously running Phase 6 Task 3A (agent weight optimisation via scipy.optimize on background compute). The two tasks don't interfere. Their outputs integrate cleanly at the next synchronisation point.

This parallelism compresses the development timeline significantly. Sequential development — research, then implementation, then validation — would take three to four times as long. Asynchronous parallel development means that validation of one feature is running while implementation of the next is underway.

What doesn't work: ambiguous ownership

The failure mode we've encountered is ambiguous ownership. When a task requires both research and implementation in rapid iteration — when the implementation reveals something that changes the research direction, which then changes the implementation — the handoff protocol breaks down. Both agents end up working on overlapping pieces with slightly different assumptions, producing inconsistent outputs that require significant reconciliation work.

The solution has been to assign a primary owner for every ambiguous task and have the secondary agent in a supporting role only. "Gemini owns Task 3A, Claude supports by setting up the scipy.optimize framework" is unambiguous. "Both agents work on the weight optimisation" is not.

What it reveals about AI collaboration

The Stack$Trader collaboration is an early data point in what will become a common pattern: AI systems with different strengths working together on complex, multi-domain problems that no single system can handle optimally. The key lesson from this project is that the collaboration is only as good as the protocol that governs it. Strong, explicit handoffs. Clear ownership. Explicit decision gates. Git as ground truth.

The romantic version of AI collaboration is two intelligences brainstorming together in real time. The reality is more like a well-run engineering organisation: clear roles, documented communication, explicit deliverables, and a shared artifact (the codebase) that neither agent can unilaterally corrupt. That structure is what makes it work.
